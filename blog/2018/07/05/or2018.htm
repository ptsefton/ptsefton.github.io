Trip Report

Date Thu 05 July 2018

I recently attended OR2018, the Open Repositories conference from June 4-7, 2018 in Bozeman Montana.

My trip was funded by the University of Technology Sydney (UTS).

Mission

Gavin Kennedey from QCIF was also in attendance, and we were on something of a mission - to promote and get feedback on the recent work we've been doing on the the ReDBox research data management platform. We ran a workshop on the Monday of the conference, in which participants got to learn about

Gavin and I also presented a general introduction to ReDBOX and the provisioner, and I went into more detail about the DataCrate standard for shipping and showing-off research data that I have been leading, with help from a growing community of supoporters. I also did a presentation in the technical session which included a live demo of using ReDBox to ingest DataCrates - showing how it could 'sniff out' metadata from DataCrate packages. As it was a more of an demo that will not translate so well to a blog post,

What else did you learn?

Open Repositories now has enough going on about Research Data Management that I was able to spend most of my time in sessions relevant to my work (when I wasn't meeting with people directly).

I heard people talk about a few things that confirmed

With some others from Australia (Vladimir Bubalo from Macquarie and the aforementioned Gavin), chasing up some more details about the Dataverse repository software. There is some interest from mq in what's available in the way of open source research data repositories. Dataverse looks like it's a thriving product and would be a good integration target for ReDBOX - but the problem they're still grappling with is large file support, which is the same issue with any monolithic repository when you try to put large volulmes of data through the API or web.

There was a really interesting talk from the UK Jisc Research Data Shared Services (RDSS) project (on which I have done some very part time consulting via Artefactual in Canada) about how they failed to get a Samvera repository working as part of their offering.

This is why, at UTS, we are building research data management systems that are distributed, with the largest data residing in large scalable file storages system, described and linked to Data Management Plans, data descriptions of data publication records. For us, the repository is a collection of services which includes the central index and a variety of stores, rather than an application.

One of things I was keen to do at OR2018 was to find out more about the Oxford Common Filesystem Layout which is an initiative of Oxford Emory, Standford, Cornell and Duraspace- it's about how you organise digital assets in the kind of deconstructed architecture I described about for UTS so that you can run services against them: index them for discovery, check their integrity, generate dissemination versions for distribution, report on items due for disposal and so on.

I couldn't get to the presentation but I got a personal run through with Neil Jacobs from Oxford who talked me through the standard and the process being used to develop it. Neil assured me we can go ahead and implement against the draft spec (and challenged me with the statement that at Oxford they don't like to talk about data vs metadata, it's all data. I'm still thinking that over Neil, I think I still believe in metadata).

This is from the draft dated 2018-06-22, a couple of weeks after the conference:

A general observation is that the contents of a digital repository -- that is, the digital files and metadata that an institution might wish to manage -- are largely stable. Once content has been accessioned, it is unlikely to change significantly over its lifetime. This is in contrast to the software applications that manage these contents, which are ephemeral, requiring constant updating and replacement. Thus, transitions between application-specific methods of file management to support software upgrades and replacement cycles can be seen as unnecessary and risky change, changing the long-term stable objects to support the short-term, ephemeral software.

By providing a specification for the file-and-folder layout on disk, the OCFL is an attempt at reducing, or even eliminating, the need for these transitions. As an application-independent specification, conforming applications will natively 'understand' the underlying file structure without needing to first transition these contents to their own format. https://ocfl.io/
